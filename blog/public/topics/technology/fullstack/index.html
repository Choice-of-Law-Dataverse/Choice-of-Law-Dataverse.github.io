How Full Stack Applications Actually Work#

Full stack applications are the backbone of modern digital experiences. They power websites, mobile apps, and online services just alike.

The term “full stack” encompasses both frontend (user-facing) and backend (server-side) components.

It is important to understand how these components interact with each other to create seamless, functional applications.

Components#

  1. Frontend The visible part users interact with is called the frontend. It is usually built using a combination of HTML, CSS, and JavaScript. Writing a whole frontend using just these languages in their raw form would be quite tedious, which is why there are plenty frameworks that have been developed on top of them. To build the Choice of Law Dataverse we use Vue.js, which is a framework built on top of JavaScript.

  2. Backend The server-side logic handling data processing, is called the backend.

  3. Database The database is the storage layer, where data is not only stored but also managed. Generally speaking, there are two types of databases: SQL and NoSQL. I will focus on SQL, since it is the most popular (and the “OG”). Basically, you can imagine SQL (which stands for “Structured Query Language”) to be a storage system for data in tabular format. There can be multiple tables and entries within tables can be linked to other entries from other tables. Maintaining a tabular format for any data that is stored, allows for an efficient as well as organized access to the data.

  4. API Layer The API (Application Programming Interface) acts as a bridge between frontend and backend, using REST or GraphQL for structured communication. For the Choice of Law Dataverse, we rely on ideas following the REST principles.

Communication Between Components#

The way the frontend talks to the backend and vice versa, is by using so called HTTP (Hypertext Transfer Protocol). You can think of HTTP as a formal set of rules how the involved computers should behave when they talk to each other. Following these set of rules ensures that the computers actually understand each other.

The idea of a RESTful API is yet another way of formalizing the communication between frontend and backend. This time, you can think of the REST-rules as the kind of language used for the communication, as it defines the data structure for exchanging information. Basically, the backend exposes endpoints that are consumed by the frontend to fetch or modify data.

The backend translates the requests from the frontend so that the database can understand and execute them. What is then returend by the database might be further processed by the backend, before it is sent back to the frontend.

Bringing the Components Together#

Deployment tools like Docker and Kubernetes are often used to bring the individual components to life. Containerization and scalability.

Git for version control and collaborative development.

CI/CD pipelines for automating code integration, testing, and deployment.

Common Full Stack Architectures#

Monolithic architecture: Single application encompassing all layers - simpler but less scalable.

Microservices architecture: Modular approach with independent services communicating over APIs.

Serverless architecture: Backend logic run as managed services in the loud, emphasizing scalability and reduced operational overhead.

Challenges in Full Stack Development#

Managing communication between components, especially when scaling applications.

Security considerations, such as protecting APIs and managing authentication.

Keeping frontend and backend synchronized during updates or feature rollouts.

Conclusion#

Full stack applications integrate various components to deliver seamless functionality.

The collaborative and iterative nature of full stack development.
